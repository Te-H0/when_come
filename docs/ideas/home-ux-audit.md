## 표지 요약 — Quick Wins (반나절 이내, 즉시 적용 권장)

> "정류장 도착 5초 룰"을 만족시키기 위해 Home은 **무엇을, 언제, 어느 방향으로** 세 가지를 즉시 보여줘야 한다. 현재 화면은 이 셋을 다 갖추고 있으나 **시각적 위계가 평탄하고, 무의미한 ID와 무반응 UI가 신뢰를 깎고 있다.** 아래 6개를 먼저 처리하면 체감 품질이 가장 크게 오른다.

| # | 항목 | 효과 | 난이도 |
|---|------|------|-------|
| QW1 | 지하철 stop의 ARS/ODsay ID 노출 제거, 버스만 ARS 표시 + "ARS" 라벨 추가 | High — 인지 부하/혼란 즉시 제거 | 5분 |
| QW2 | "다음 교통수단" 카드 우측 ChevronRight 제거 (또는 onClick 부여) — affordance 정합성 | High — 거짓 신호 제거 | 10분 |
| QW3 | "탑승 완료" 클릭 후 toast + "되돌리기" 액션 (`sonner` 이미 있음) | High — 실수 안전망, 출퇴근 환경 필수 | 30분 |
| QW4 | 경로 타임라인의 "버스/전철" 텍스트를 **호선 번호/색 칩**으로 교체 (`5524`, `2호선` 등) | High — 5초 룰의 핵심, 환승 정보 즉시 인식 | 1~2시간 |
| QW5 | 도착 시간 두 줄(`3분 56초 후` / `9분 56초 후`)에 **"이번/다음" 라벨** 추가 또는 시각 차이로 위계 부여 | Med — "두 줄의 의미"가 즉시 전달 | 30분 |
| QW6 | 헤더 경로 이름 옆 "활성 경로 1/2" 같은 카운터 또는 탭 화살표 affordance — 선택지 인지 | Med | 30분 |

**중기 (1~2일):** 출/퇴근 자동 전환, 경로 이름 자동 추천, 도착 임박 시 햅틱·시각 강조, 빈 상태 메시지 다양화.

**장기 (별도 PRD):** 출퇴근 경로 메타(출근/퇴근/기타) 도입, 위치/시간 기반 자동 활성 경로 선택, 위젯/잠금화면, 환승 안내 카드.

## 2. 우리 앱 특성 vs 현재 UI 갭 분석

### 2.1 6개 원칙별 평가

| **패턴 인식** (평소와 다름이 보여야) | ★☆☆☆☆ | "평소 3분인데 오늘 8분"같은 신호 없음. 도착시간만 절대값으로 보여주고 baseline 없음. 다만 이건 장기 작업 |

### 2.2 가장 큰 갭 3가지 (우선순위)

1. **시각적 우선순위 평탄 + 5초 룰 미달** — 목적지/타임라인이 자리만 차지하고 핵심 정보가 후순위. 한국 대중교통 특성상 "환승역/호선 번호"가 행동을 결정하는데 타임라인이 이를 표시 안 함. **→ QW4 + D5**
2. **신뢰 부족 (자동 갱신 + 신선도 표시 없음)** — 출퇴근 앱은 시간이 곧 가치. 사용자가 "이거 5분 전 데이터 아니야?" 의심하면 앱을 안 씀. **→ D7 + D8**
3. **자동화 부재 (출/퇴근 경로 자동 선택 없음)** — "매일 같은 경로 반복"이 핵심인데 매일 직접 토글. 첫 사용자 경험에서 가장 큰 "어, 이거 똑똑하지 않네" 순간. **→ D9**

## 5. 비-UI 차원의 발견 (도메인 모델/데이터 흐름)

UX는 결국 **무엇을 알 수 있는가**의 문제다. 현재 도메인 모델에서 부족한 것 정리.

### 5.1 경로 메타 부족

`SavedRoute` 타입(`mockData.ts:33-40`)이 가진 정보:

```ts
{ id, name, from, to, segments, isActive }
```

**부족한 것:**

| 필드 | 사유 |
|------|------|
| `purpose: 'commute_to_work' \| 'commute_home' \| 'other'` | 시간대 자동 선택, 이름 자동 추천, "퇴근 경로로 전환" CTA의 전제 |
| `preferredTimes?: { weekday: number[]; startHour: number; endHour: number }` | 출근 시간 가변 사용자 (예: 평일 08–10시, 휴일 미사용) |
| `lastUsedAt?: string` | "최근 안 쓴 경로" 정리 제안, 패턴 인식 |
| `estimatedDurationMin?: number` | "평소 N분 걸리던 경로" 비교 baseline |

→ **B-B1 작업으로 `purpose`만 먼저 도입하고 나머지는 LT에 보류.**

### 5.2 도착 데이터 신선도 메타 없음

현재 `ArrivalData`는 raw items만 가짐. **언제 가져왔는지** 정보가 없어 UI에서 보여줄 수 없음.

```ts
// 현재
type ArrivalData = { type: 'subway'; items: [...] } | ...

// 제안
type ArrivalData = {
  type: 'subway' | 'bus' | 'odsay'
  items: [...]
  fetchedAt: number  // Date.now()
  source: 'realtime' | 'cache' | 'fallback'
}
```

`fetchedAtRef`로 컴포넌트가 따로 추적하고 있지만(`Home.tsx:66-69`), 데이터에 묶어두면 신뢰 라벨 표시가 단순해짐.

### 5.3 진행 상태(currentSegmentIndex)가 컴포넌트 useState

새로고침/앱 재진입 시 진행 상태 사라짐(0으로 reset). 출퇴근 중 화면 끄고 켰을 때 "어디였지?"가 됨.

→ localStorage 또는 더 나아가 서버 측에 "현재 진행 중인 경로/세그먼트" 메타 저장. 단, **자동 진행(GPS) 도입과 함께 설계해야** 중복 작업 안 됨.

### 5.4 출/퇴근 자동 감지의 데이터 비용

- 시간대 기반: 추가 비용 0
- GPS 기반: 권한 + 배터리 + 백그라운드 정확도. **PWA 한계 큼** — 별도 네이티브/위젯 검토 필요

→ Phase B는 시간대 기반(LT 가성비 높음), Phase C에서 GPS 검토.

### 5.5 환승 정보의 정의가 모호

현재 segment는 "정류장/역" 단위. "환승"이라는 1급 객체가 없음. 카드에는 segment를 나열만 함.

- 환승 시간 추정 (역 도보 N분) 표시 어려움
- "환승 후 어느 호선" 강조 어려움 (D1과 연관)

→ 데이터 모델보다 **UI 렌더 시 segment[i].stop ↔ segment[i+1].stop의 관계를 "환승"으로 해석**하는 함수 도입(`getTransferInfo(prev, next)`). DB 모델 변경 없이 가능.

### 5.6 사용자 행동 데이터 0

"매일 같은 패턴"을 강조하는데 정작 그 패턴을 학습할 데이터가 없음:

- 어느 시간에 앱을 켜는지
- 어느 segment에서 "탑승 완료"를 누르는지
- 새로고침을 얼마나 자주 하는지(불안의 신호)

→ **개인정보 부담 적은 익명 이벤트 1~2개**부터 시작 권장. 단, PRD 필요.

## 6. 종합 권고 (실행 순서)

1. **이번 주: Phase A 8개 묶음 1~2 PR로 머지.** 사용자 보고 3건 + 시각적 위계 + 신뢰 게이지 한 번에 해결. 솔로 파운더 시간 효율 최고.
2. **다음 주: B-B1, B-B2, B-B3 묶음 — 자동화의 첫 걸음.** purpose 도입은 ADR을 1장 쓰고 가는 게 좋음. 미리 모델 잡아두면 LT1(GPS)·LT4(패턴) 모두 위에 올라간다.
3. **이후: B-B5, B-B6 — 신뢰·임박감.** 사용자 인터뷰 1~2명 거친 후 결정.
4. **장기: LT 시리즈는 사용자 수가 늘었을 때.** 그 전엔 위 두 단계로 충분히 좋다.

핵심 메시지 한 줄: **"평탄한 위계와 무반응 UI를 정리하는 것만으로도 출퇴근 5초 룰을 거의 만족시킬 수 있다. 진짜 도약은 자동화(출/퇴근 자동 전환)에 있다."**
